# Stance

Stance is a professional communication tool that generates three ready-to-send reply variations for difficult workplace conversations. You paste the message you received, choose your situation (e.g. unpaid invoice, salary negotiation, scope creep) and desired outcome, optionally add your writing style examples and role context, and hit Generate. The app sends your input to a Cloudflare Worker, which calls the Groq API (Llama 3.3 70B) and returns three labelled reply cards — Firm, Diplomatic, Brief — each editable via quick-action refinements.

---

## Features

- **3 reply variations per generation** — labelled Firm, Diplomatic, and Brief, generated by Llama 3.3 70B via Groq
- **14 situation types** — 5 shown by default, 9 more behind an expand toggle (payment chase, salary negotiation, scope creep, resignation, cold outreach, etc.)
- **12 desired outcomes** — e.g. get a response, resolve professionally, set boundaries, escalate urgency, buy time
- **Writing style examples** — paste samples of your own writing; the prompt instructs the model to match your tone and vocabulary
- **Optional email thread context** — attach the broader thread so the model can write with full conversation awareness
- **Optional role field** — tell the model your role (e.g. Freelancer, Business Owner) for contextually appropriate replies
- **Quick-action refinements** — per-card actions: Make Firmer, More Polite, Make Shorter, More Human, More Confident; each sends a new generation request
- **Playbooks** — 3 built-in templates (Unpaid Invoice, Scope Creep, Client Complaint); up to 3 custom playbooks saved in localStorage; load any playbook to pre-fill the workspace
- **Resizable split panel** — draggable divider between input panel and reply grid; width persists in localStorage
- **Mobile layout** — stacked single-column on screens below md breakpoint
- **Light/dark theme toggle** — persisted in localStorage
- **Rate limiting** — 10 generations per IP per day, enforced in the Cloudflare Worker via KV
- **Umami analytics** — tracks `generate_replies` events with situation type and whether writing examples were provided

## Local Development

### 1. Frontend

```bash
# Install dependencies
npm install

# Start dev server (http://localhost:5173)
npm run dev
```

You need `VITE_WORKER_URL` set in a `.env` file pointing at either the deployed worker or the local worker below.

```bash
# .env
VITE_WORKER_URL=http://localhost:8787
```

### 2. Cloudflare Worker

```bash
cd stance-worker

# Install worker dependencies
npm install

# Start worker locally (http://localhost:8787)
npm run dev
```

The local worker needs the Groq API key available. Set it as a secret or pass it as an environment variable during local dev:

```bash
# For local dev only — wrangler will prompt or you can set it in .dev.vars
echo "GROQ_API_KEY=your_key_here" > .dev.vars
```

`RATE_LIMIT_KV` is bound in `wrangler.jsonc`. When running locally with `wrangler dev`, KV reads/writes use an in-memory store by default — no remote KV access needed.

## Environment Variables

### Frontend (`.env`)

| Variable | Required | Description |
|---|---|---|
| `VITE_WORKER_URL` | Yes | Full URL of the deployed Cloudflare Worker (e.g. `https://stance-worker.your-subdomain.workers.dev`). The frontend POSTs all generation requests to this URL. |

### Cloudflare Worker (secrets via Wrangler)

| Variable | Required | Description |
|---|---|---|
| `GROQ_API_KEY` | Yes | API key for the Groq API. Get one free at [console.groq.com](https://console.groq.com). Set as a Worker secret: `wrangler secret put GROQ_API_KEY` |

The KV namespace (`RATE_LIMIT_KV`) is a Cloudflare binding, not an environment variable. Its ID is declared in `wrangler.jsonc` and provisioned in your Cloudflare account.

## Deployment

Deploy the worker first so you have its URL to set on the frontend.

### 1. Deploy the Cloudflare Worker

```bash
cd stance-worker

# Authenticate with Cloudflare (one-time)
npx wrangler login

# Create the KV namespace if not already created
npx wrangler kv namespace create RATE_LIMIT_KV
# Copy the returned ID into wrangler.jsonc under kv_namespaces[0].id

# Set the Groq API key as a secret
npx wrangler secret put GROQ_API_KEY

# Deploy
npm run deploy
```

The worker URL will be printed on deploy: `https://stance-worker.<subdomain>.workers.dev`

### 2. Deploy the Frontend to Vercel

```bash
# Install Vercel CLI if needed
npm install -g vercel

vercel
```

During setup or in the Vercel dashboard, add the environment variable:

```
VITE_WORKER_URL = https://stance-worker.<subdomain>.workers.dev
```

Vercel will run `tsc && vite build` and serve the `dist/` output.

## How the AI Pipeline Works

```
User input (browser)
  └─ incomingMessage (required)
  └─ situation (required)
  └─ desiredOutcome
  └─ role (optional)
  └─ writingExamples[] (optional)
  └─ threadContext (optional)
        │
        ▼
aiService.generateReplies()
  POST VITE_WORKER_URL with JSON body
        │
        ▼
Cloudflare Worker (stance-worker/src/index.js)
  1. CORS preflight check
  2. Parse request body
  3. Rate limit check:
       key = "ratelimit:{CF-Connecting-IP}:{YYYY-MM-DD}"
       read count from RATE_LIMIT_KV
       if count >= 10 → return 429 { error: "daily_limit_reached" }
       increment count, write back with 90,000s TTL (25h)
  4. Build prompt:
       system: "Generate exactly 3 reply variations as JSON {replies:[{label,content}]}"
       user:   role + situation + outcome + incoming message + writing examples
  5. POST to Groq API
       model: llama-3.3-70b-versatile
       temperature: 0.7
       response_format: { type: "json_object" }
  6. Parse Groq response → return { replies: [{label, content}] }
        │
        ▼
useGeneration hook
  Maps replies: label → title
  Assigns stable IDs (reply-{timestamp}-{index})
  Sets replies[] in state
        │
        ▼
ReplyGrid → ReplyCard × 3
  Each card shows: title (Firm / Diplomatic / Brief), content, copy button
  Quick actions (Firmer, More Polite, Shorter, More Human, More Confident)
    → call generateReplies() again with the card content as the incomingMessage
    → replace card in-place with refined version
```

## Rate Limiting

The Cloudflare Worker enforces a **10 requests per IP per day** limit using Cloudflare KV.

- **Key format:** `ratelimit:{CF-Connecting-IP}:{YYYY-MM-DD}` (date in UTC)
- **Limit:** 10 requests. On the 11th request, the worker returns `HTTP 429` with `{ "error": "daily_limit_reached" }`.
- **TTL:** Each KV entry expires after **90,000 seconds (25 hours)** to handle midnight edge cases cleanly.
- **Frontend handling:** `aiService.ts` detects the `daily_limit_reached` error code and shows the message: *"You've used your 10 free replies for today. Your limit resets at midnight. Come back tomorrow."*
- **Counter increment:** Written with `ctx.waitUntil()` so the response is not delayed by the KV write.
